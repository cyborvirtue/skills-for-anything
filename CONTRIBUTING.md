# 参与贡献

<p align="right">
  简体中文 | <a href="./CONTRIBUTING_EN.md">English</a>
</p>

感谢你帮助完善 `Skills for Anything`。

这是一个精选列表。目标不是尽可能收集更多仓库，而是维护一个高信号、便于发现优质资源的 AI Skills 导航项目。

## 我们接受什么类型的提交

欢迎提交以下类型的内容：

- 可复用的 skill 或 skill pack
- skills marketplace 或安装器
- skills registry 或跨平台分发工具
- skills 导航型目录
- 大仓库里特别值得单独发现的 skill 目录

更强的候选条目通常具备以下至少一项特征：

- 有明确的复用价值
- 持续维护中
- 有一定关注度或采用信号
- 领域定位清晰
- 对整个 skills 生态有明确补充价值

## 我们不接受什么

请不要提交以下内容：

- 通用 prompt 合集
- 与 skills 关系很弱的 AI 工具导航
- 以文章、博客、教程为主的内容
- 已经停止维护且价值不明确的仓库
- 只是把 `skill` 当作营销词，但并没有真实可复用 skill 资产的项目

## 收录范围规则

这个列表遵循三条基本原则：

- GitHub-first
- 按领域组织
- 精选优先，而不是穷举优先

这意味着：

- 默认优先提交 GitHub 仓库
- 只有当某个具体 skill 目录本身比整个仓库更值得发现时，才直接链接到该目录
- 不会为了让每个分类看起来平均，而故意塞入弱条目

## 条目格式

`README.md` 中统一使用这种格式：

```md
- [project-name](https://github.com/owner/repo) - 一句话说明它为什么值得关注。
```

如果你提交的是某个具体 skill 目录，则使用：

```md
- [skill-name](https://github.com/owner/repo/tree/main/path/to/skill) - 一句话说明它为什么值得关注。
```

写描述时请注意：

- 尽量保持一句话
- 语言具体，避免空泛
- 不要使用 “best”、“ultimate”、“revolutionary” 这一类夸张表达
- 重点写“它为什么有用”，而不是仅仅解释名字

## 在提 PR 前请先检查

- 链接可正常访问
- 仓库中尚未重复收录
- 分类放置合理
- 描述简洁且准确
- 这条提交确实提升了列表的发现价值

## 建议的 PR 内容

建议在 PR 描述中包含：

- 提交的 URL
- 你建议的分类
- 一句话描述
- 1 到 2 句说明它为什么应该出现在这里

## 审核标准

维护者可能会拒绝以下条目：

- 会削弱列表主题聚焦度的内容
- 与现有更强条目重复的内容
- 还太早期、太模糊或信号太弱的内容

这里更看重质量，而不是分类上的平均分布。有些领域本来就比其他领域更成熟。
